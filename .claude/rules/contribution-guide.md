# .claude/rules/ の変更ガイドライン

## この節が対象とする範囲

`scripts/validators/` 配下のバリデーターは、プラグイン定義ファイル（frontmatter、`hooks.json`、`.mcp.json`、`.lsp.json`、`plugin.json`、`marketplace.json`、`monitors.json` 等）の静的な構造・スキーマのみを検証できる。以下の「新規定義を追加する場合」「既存定義を更新する場合」の同期契約は、この**静的に検証可能なフィールド・スキーマの変更**（フィールドの追加・削除・有効値の変更など）に限定して適用される。

一方、`.claude/rules/` の記述の多くは、フックの発火タイミング、フックスクリプトが受け取る/返すJSONの意味、HTTPフックのリクエスト/レスポンス処理の詳細といった**ランタイム挙動の説明**であり、これらに対応するバリデーターは存在せず、静的ファイル検証の性質上存在させることもできない。ランタイム挙動の記述のみを変更する場合は、以下の「バリデーター/テストの同期」を求めない。

## 新規定義を追加する場合

ドキュメント作成だけでなく、対応するバリデーターとテストも追加すること（対象は静的に検証可能なスキーマ・フィールドの新規追加）：

1. `.claude/rules/xxx.md` - 仕様ドキュメント作成
2. `scripts/validators/xxx.py` - バリデーター作成
3. `scripts/tests/test_xxx.py` - テスト作成
4. `scripts/validators/__init__.py` - エクスポート追加
5. `scripts/validate_plugin.py` - パス検出ロジック追加
6. 関連する既存ドキュメントからのリンク追加（例: `plugin-manifest.md`）

詳細は `scripts/README.md` の「新しいバリデーターの追加方法」を参照。

## 既存定義を更新する場合

静的に検証可能なフィールド・スキーマの変更については、ドキュメントの変更に合わせて対応するバリデーターとテストも必ず同期すること：

- フィールドの有効値を変更 → バリデーターの許可リストを更新
- 新しいフィールドを追加 → バリデーターに検証ロジックを追加
- 新しいイベント/オプションを追加 → バリデーターの許可リストに追加
- 上記すべてに対応するテストを追加・更新

なお、フックの発火タイミングやHTTPフックのレスポンス処理など、対応するバリデーターが存在しないランタイム挙動の記述を更新する場合は、上記の同期は不要（該当するバリデーター/テスト自体が存在しないため）。

## Claude Code アップデート時の追跡方針

このリポジトリはプラグイン作成・更新時に参照するリファレンスであり、Claude Codeの全変更履歴を追跡する場所ではない。アップデート対応時は以下の基準で取捨選択すること：

**追跡する（ドキュメントに反映する）:**

- プラグイン定義ファイル（frontmatter）のフィールド追加・変更・削除
- `hooks.json` のスキーマ変更（新イベント、新フックタイプ、新フィールド）
- プラグインのフックスクリプトが出力するJSON（`permissionDecision` 等）の挙動変更
- `marketplace.json` / `plugin.json` のソース指定やフィールド変更
- プラグイン定義の書き方に影響するバグ修正（バリデーション挙動の変更等）

**追跡しない（ドキュメントに反映不要）:**

- ランタイムのバグ修正（例: フックの発火タイミング修正、セッション切り替え時の挙動修正など）。プラグイン定義の書き方に影響しない内部修正は記載しない
- ランタイムのツール呼び出しAPIの変更（例: `Agent()` や `SendMessage()` のパラメータ変更）
- プラグイン定義に影響しないCLI挙動の変更
- プラグインが利用しないfrontmatterフィールドの削除（既存プラグインで未使用の場合）
- Claude Code内部のUI/UX変更

**Issue作成時のトリアージ義務:**

CHANGELOGの変更からIssueを作成する際は、検出した各変更について上記の基準で追跡対象/対象外を判定し、判定結果をチェックリスト形式でIssue本文に含めること。追跡対象外と判定した変更はIssueの対応内容に含めない。

```markdown
## トリアージ結果

### 追跡対象
- [ ] 変更内容（該当基準: ○○）

### 追跡対象外
- 変更内容（理由: ○○）
```

## ドキュメントの記述方針

ドキュメントには現在の仕様を記載する。バグの経緯（「以前は○○というバグがあったが修正された」等）は記載しない。プラグイン作者が知るべきは現在の挙動であり、過去のバグではない。

### バージョン注記を書かない

原則として、機能の導入バージョンや変更・バグ修正の時期を示す版数注記は記載しない。具体的には以下をすべて避ける：

- 機能の最低必要バージョン注記：「（v2.1.69以降）」「【v2.1.63以降】」「v2.1.84で追加」など
- 変更・バグ修正の経緯：「v2.1.39での変更」「v2.1.74以前は○○だった」「○○されるようになりました」「破壊的変更」など

理由：このリポジトリは「今プラグインを書く」ためのリファレンスであり、プラグイン作者が必要とするのは**現行の挙動**である。Claude Code は頻繁に自動更新されるため最低必要バージョンの実務的価値は低く、版数注記はドリフトして陳腐化・誤記の温床になる（実際に複数の版数誤りが発生していた）。各ドキュメントは「現在こう動く」という現行仕様のみを記述する。

**例外**：互換性上の注意が実務的に本当に必要な場合に限り、本文中に「最低必要 Claude Code バージョン: vX.X.X」と明示的に記載してよい。その場合も CHANGELOG で実バージョンを必ず確認すること（CHANGELOG はバージョンを飛ばすことがある。例: 2.1.63 の次は 2.1.66 で、2.1.64/2.1.65 は存在しない）。
