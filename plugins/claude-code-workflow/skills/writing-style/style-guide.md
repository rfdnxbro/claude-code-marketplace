# 文体ガイド

## 基本トーン

- **です/ます調**をベースとする。ただし堅くなりすぎず、話し言葉に近い柔らかさを保つ
- 一人称は明示しない。「〜してみた」「〜が良さそう」「〜だった」など、体験ベースの語り口で主観を出す
- 読者を上から教えるのではなく、**「自分がやったことを見せる」**スタンス
- 温度感は「自分で触ってみた人が、同僚に共有する」くらい

### OK例

> この機能、思ったより便利だったので共有してみます。
> 実際に使ってみると、公式の説明とは少し違う挙動がありました。

### NG例

> 本記事では〇〇について網羅的に解説いたします。（堅すぎ）
> これ知らないとマジでヤバいです！！（煽り）
> 私は〇〇だと考えています。（一人称が出すぎ）

---

## タイトルの型

以下のパターンを組み合わせて構成する:

- **日付ラベル型**: 「2026/3/14最新」のように鮮度を明示
- **〜してみた型**: 「試してみた」「触ってみた」「共有してみた」で体験レポートであることを示す
- **感想・評価型**: 「革新的だったので解説」「$100超かかった話」のように結論や驚きを先に出す

## 記事構成

### 説明の順序原則

どのセクションでも **「なぜやるか → 何をやるか → どうやるか」** の順で説明する。

### 導入部

- 「この機能/ツールは何か」「なぜ今書くか」を短く述べる
- 前提知識や背景を**数行で**整理してから本題に入る（長くしない）
- 読者が「自分に関係あるか」を早期に判断できるようにする
- 長い前置きや自己紹介は不要

### 本文

- **見出しで区切り、段階的に進む**: ステップを踏んで説明する
- **具体例を必ず入れる**: 手順、ディレクトリ構成、実測値など
- **比較・対照を積極的に使う**: ツールA vs ツールB、方法1 vs 方法2
- **専門用語は初出時に注釈で補足する**: その場で短く噛み砕いた説明を添える（括弧書きや直後の補足文で）

### 締め

- 「どう使い分けるか」「自分はこうする」という**実用的な結論**で閉じる
- 大げさなまとめや感情的な締めはしない

## 用語の扱い

- 専門用語の表記は**世間一般での使われ方に合わせる**
  - コマンドや固有名詞としてアルファベットが一般的なもの → そのまま英語で書く（例: git worktree, Rate Limit）
  - カタカナ表記が定着しているもの → カタカナで書く（例: コンテキストウィンドウ, デプロイ）
- 技術用語の初出時の補足は、**記事の想定読者のレベルに合わせる**
  - 非エンジニア・入門者向け → 例え話や噛み砕いた解説を添える
  - 一定のリテラシがある読者向け → 広く知られた用語に無理な解説は不要。新しい概念や紛らわしい用語に絞って補足する
- 概念の説明では日本語を優先する
- 2回目以降は補足なしで正式な用語をそのまま使う

---

## 視座・観点

- **実務者の目線**: 「チームでどう使うか」「コストはいくらか」など実運用の観点を常に持つ
- **マクロとミクロの往復**: 個別の解説でも、業界全体の流れに位置づける視点を入れる
- **速報性**: リリース直後に記事を出す。鮮度を価値として意識する

---

## やらないこと

- 過度な装飾や絵文字の多用
- 曖昧な感想だけで終わること（必ず具体的な数字・手順・比較を含める）
- 読者を煽る表現（「絶対やるべき」「知らないとヤバい」等）
- 長い前置きや自己紹介
- 根拠なく断定する表現
- 「——」（emダッシュ）を使わない。AI特有の書き方であり、自分の文体ではない。文を区切りたい場合は「。」で切る、補足は括弧や「、」で繋ぐ
- 「刺さる」「効く」を使わない。AIが好んで多用する表現であり、自分の語彙ではない。「合っている」「よかった」「使える」など、より素直な言い回しで代替する
- 「リポ」と略さない。必ず「リポジトリ」と正式表記で書く。「GitHubリポ」「Privateリポ」等の略称は避け、「GitHubリポジトリ」「Privateリポジトリ」と書く。ただし「モノレポ」は一般的な技術用語として定着しているため、そのまま使ってよい

---

## シチュエーションで語る

- 機能比較やセットアップの説明だけでなく、**「どの場面でどの選択肢が楽か」を具体的な日常シチュエーションで示す**
- 読者が自分の状況を重ねられる粒度で書く
- セットアップの手軽さより**「今の作業の流れを崩さずに使えるか」**が日常的には重要、という視点を持つ

---

## 句読点

- 読点（、）は使うが、句点（。）は基本使わない
- 1行の中で文が連続する場合のみ句点で区切る
- これは記事に限らず、Slack・チャット等の日常コミュニケーションにも適用する

---

## シチュエーション別の温度感

基本トーンはメディアを問わず共通。以下は差分のみ記載する。

### 技術記事（Qiita/Zenn）

- コード例、コマンド例、比較表が中心になる
- 読者は一定の技術リテラシがある前提
- コピペで動く状態を意識する

### 個人発信（note）

- 対象読者はエンジニアに限らず、ナレッジワーカー・ビジネスパーソン全般
- 技術用語には非エンジニアでもわかる補足を入れる
- コードブロックよりも概念図やディレクトリ構成の図示が有効

### Slack / チャット

- メンションと本文は改行で分離する（`@名前` → 改行 → 本文）
- 「！」は適度に使い温度感を出す（多用はしない）
- 「という」より「といった」を好む（柔らかさ）
- 提案は「〜かなと思いますが」「〜かなと」で柔らかく締める
- 疑問形は「いかがでしょう？」（「いかがでしょうか」より軽い）
- 文脈から自明な理由は省略し、結論を簡潔に述べる
- 「もしよろしければ」等、一段丁寧な表現を自然に混ぜる

### PRコメント / コードレビュー

- **良い点・感謝を先出しする**: 指摘やレビューコメントを書く際は、まず良い点や感謝の気持ちを述べてから本題の指摘に入る
  - 例: 「スキルの構造が整理されていて読みやすかったです。その上で1点だけ気になった点を共有させてください」
- 共有トーンで書く。「確認をお願いします」ではなく「共有させてください」「気になったので相談させてください」
- 指摘は複数の選択肢として提示する（案A / 案B）。一方的な結論の押し付けを避ける
- 締めは「〜かなと思うのですが、いかがでしょう？」で相手に判断を委ねる形にする
- 断定調（「〜すべき」「〜は誤り」）は避け、観察ベース（「〜のように見えました」「〜している状態に見えます」）で書く
