# 文体ガイド適用スキル

ユーザー名義で外部に出る日本語のプローズを書くとき、ここに定義された文体ルールに従う。

## 発火条件

以下のいずれかを書いている／推敲している／レビューしているとき:

- Qiita/Zenn の技術記事
- note の記事（ナレッジワーカー・ビジネスパーソン向け）
- Slack メッセージの下書き（自分発信のもの）
- ユーザーが書く PR コメント・コードレビューコメント
- MENTA メンタリングの返信文面
- その他、ユーザー本人の声として外部に出る日本語の長文

逆に、以下では発火しない:

- コード本体・コミットメッセージ・ログ出力・CLI 出力
- 機械的な定型文（テンプレ、自動生成のサマリ）
- 他者が書いた文章のレビュー・校正（他人の PR コメント、他人の記事のレビュー等）。文体は本人のものなので矯正しない
- ユーザーが「文体は気にしないで」と明示している場面

## 手順

1. **文体ガイド本体を読み込む**（Read ツールで以下 3 ファイルを 1 メッセージ内で並列取得）
   - `${CLAUDE_SKILL_DIR}/style-guide.md` — 基本トーン・タイトル型・記事構成・用語の扱い・句読点・シチュエーション別温度感・禁止事項
   - `${CLAUDE_SKILL_DIR}/glossary.md` — 表記揺れ辞書（統一表記の正/誤）
   - `${CLAUDE_SKILL_DIR}/editing-process.md` — ドキュメント編集時のプロセス原則（1項目ずつ確認、実データベース）

2. **シチュエーションを特定する**
   - 媒体（Qiita/Zenn/note/Slack/PR/その他）を確認し、style-guide.md の「シチュエーション別の温度感」セクションの該当差分を適用する
   - 媒体が曖昧な場合はユーザーに確認する

3. **文章を組み立てる／推敲する**
   - 基本トーンと句読点ルールを徹底する（句点「。」は基本使わない、em ダッシュ禁止）
   - 禁止表現（「刺さる」「効く」「リポ」略称、煽り表現、長い前置き）が混入していないかチェックする
   - 用語表記は glossary.md の正表記に揃える
   - editing-process.md に従い、長い文章のブラッシュアップは1項目ずつ確認しながら進める

4. **書き終わったら自己チェック**
   - em ダッシュ「——」が紛れ込んでいないか
   - AI が好む語彙（「刺さる」「効く」「〜することができます」等）が残っていないか
   - 句点「。」が必要以上に使われていないか
   - 一人称（「私」）が前面に出すぎていないか

## エフォートレベル別の動作

現在のエフォートレベル: !`echo ${CLAUDE_EFFORT:-high}`

- `low`: 禁止表現・用語統一チェックのみ
- `medium`: 文体チェック＋句読点ルール適用
- `high`/`xhigh`/`max`: 全ガイドファイルを読み込み完全適用

## 注意事項

- ガイドの更新は、このスキルディレクトリ内（`${CLAUDE_SKILL_DIR}/style-guide.md` 等）のファイルを直接編集する
- 表記揺れ辞書（glossary.md）は随時追記する性質。新しく統一すべき表記が出てきたら追記提案も行う
- ガイドの内容自体に疑問・改訂提案が出た場合は、スキル発動とは別の文脈として扱い、ガイドファイルの改訂作業に切り替える
